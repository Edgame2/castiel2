# Changelog

All notable changes to this module will be documented in this file.

## [Unreleased]

### Added
- **Forecast period scenarios (UI contract):** `GET /api/v1/forecasts/:period/scenarios` returns `{ scenarios: [] }` with auth and tenant enforcement. Supports the analytics forecast period page (`/analytics/forecast/[period]`). Response can be extended later with real scenario data.
- **Phase 11 – Risk outcome → CAIS (record-outcome on opportunity close):** When an opportunity closes (IsClosed, CloseDate in last 24h), risk-analytics publishes `opportunity.outcome.recorded` (Plan §904). When the tenant has `outcomeSyncToCais` enabled (GET `/api/v1/adaptive-learning/tenant-config/:tenantId`), risk-analytics also calls adaptive-learning POST `/api/v1/adaptive-learning/outcomes/record-outcome` with `predictionId` = last risk evaluation id for the opportunity (from `risk_evaluations`), `outcomeValue` (1 won / 0 lost), `outcomeType` (success/failure), and `context: { opportunityId, closeDate, amount }`. Sync is non-blocking; if evaluationId cannot be resolved or adaptive-learning is unavailable, the event is still published. `generateServiceTokenForConsumer` in `src/utils/serviceToken.ts` generates a service JWT for outbound calls when Fastify is not available (event consumer). Config: `services.adaptive_learning.url` (already present).
- **W10 – Tenant ML Configuration (REQUIREMENTS_GAP_ANALYSIS Gap 4):** Cosmos container `tenant_ml_config` (risk_tenant_ml_config), partitionKey tenantId. Types: `TenantMLConfiguration`, `TenantMLConfigurationDocument`, `RiskTolerance`, `DecisionPreferences`, `ModelPreferences`, `CustomFeature`, `UpsertTenantMLConfigBody` in `tenant-ml-config.types.ts`. `TenantMLConfigService`: `getByTenantId(tenantId)`, `upsert(tenantId, body)`. APIs: `GET /api/v1/tenant-ml-config` (returns 404 if not configured), `PUT /api/v1/tenant-ml-config` (body: riskTolerance, decisionPreferences, modelPreferences, customFeatures?; tenantId from X-Tenant-ID).
- **W10 – HITL consumption of tenant ML config (Layer 6):** `RiskEvaluationService` accepts optional `TenantMLConfigService` (instantiated internally when not provided). After `risk.evaluated`, HITL logic uses tenant config when present: `riskTolerance.autoEscalationThreshold` overrides `thresholds.hitl_risk_min`; `decisionPreferences.requireApprovalForActions` gates publishing `hitl.approval.requested` (default true when no config). Routes pass `TenantMLConfigService` into `RiskEvaluationService`.
- **W9 – Reactivation evaluation and events (Layer 6, FR-6.7):** Config `services.llm_service.url` (env `LLM_SERVICE_URL`, default http://localhost:3062). `ReactivationService(fastify)`: `evaluateReactivationOpportunities(tenantId, opportunityIds, options)` calls ml-service for dormant features and reactivation prediction per opportunity, filters by minProbability (default 0.3), optionally calls llm-service for strategy when includeStrategy and llm_service.url; publishes `reactivation.opportunity.identified` for each viable opportunity, `reactivation.strategy.generated` when strategy is included. Events: `publishReactivationOpportunityIdentified`, `publishReactivationStrategyGenerated` in RiskAnalyticsEventPublisher. `POST /api/v1/reactivation/evaluate` (body: opportunityIds, minProbability?, maxOpportunities?, includeStrategy?) returns list of { opportunityId, dormantFeatures, reactivationPrediction, reactivationStrategy? }.
- **W8 – Sales Methodology (REQUIREMENTS_GAP_ANALYSIS Gap 2):** Cosmos container `sales_methodology` (risk_sales_methodology), partitionKey tenantId. Types: `SalesMethodology`, `SalesMethodologyDocument`, `MethodologyStage`, `StageRequirement`, `StageExitCriteria`, `RequiredField`, `MethodologyRisk` in `sales-methodology.types.ts`. `SalesMethodologyService`: `getByTenantId(tenantId)`, `upsert(tenantId, body)`. APIs: `GET /api/v1/sales-methodology` (returns 404 if not configured), `PUT /api/v1/sales-methodology` (body: methodologyType, stages, requiredFields?, risks?; tenantId from X-Tenant-ID).
- **W8 Layer 6 – makeMethodologyDecisions (FR-6.6):** `DecisionSource` extended with `methodology`. Types: `MethodologyFeaturesInput`, `MakeMethodologyDecisionRequest` in `decision.types.ts`. `DecisionEngineService` accepts optional `SalesMethodologyService` and `getMethodologyFeatures(tenantId, opportunityId)` (fetches from ml-service). `makeMethodologyDecisions(tenantId, opportunityId)`: fetches methodology features from ml-service, builds actions for stage requirements < 80%, stage duration anomaly, methodology fields < 70%, MEDDIC score < 0.6; persists one Decision with decisionType `methodology`, source `methodology`. `POST /api/v1/decisions/methodology` (body: opportunityId) returns decision or 204 when no findings or ml-service unavailable.
- **W7 Gap 1 – Risk Catalog–driven decisions (Layer 6, FR-6.5):** `DecisionEngineService` accepts optional `RiskCatalogService` in constructor. `Rule` has optional `catalogRiskId`. New methods: `getRulesForCatalogRisks(tenantId, catalogRiskIds)`, `mapRisksToCatalog(detectedRisks, catalog)`, `applyRiskCatalogRules(request, tenantId)` (get catalog, map risks, get rules for catalog risks, evaluate, persist decision). Types: `MappedRisk`, `ApplyCatalogRulesRequest`. `POST /api/v1/decisions/apply-catalog-rules` (body: opportunityId, detectedRisks?, riskScore?, industry?) returns decision or 204 when no catalog rules applied.

### Changed
- **anomaly.detected (Plan §7.2):** `publishAnomalyDetected` includes `tenantId` in the payload for notification-manager; accepts optional `ownerId` (opportunity OwnerId). Callers should pass `ownerId` when available so notification-manager can notify the owner. logs-events.md updated.
- **anomaly.detected ownerId from shard (Plan §7.2):** `POST /api/v1/opportunities/:opportunityId/anomalies/detect` fetches `OwnerId` from c_opportunity via shard-manager when `services.shard_manager.url` is set and passes it to `runStatisticalDetection` and `persistAndPublishMLAnomaly`; `anomaly.detected` includes `ownerId` when shard-manager is configured.

### Fixed
- **BatchJobWorker (Plan §9.3, FIRST_STEPS §5):** Removed requirement for `data_lake.connection_string` to start the worker. outcome-sync and industry-benchmarks run without Data Lake; `data_lake` is required only at run-time when `risk-snapshot-backfill` runs (throws if unset). Enables outcome-sync in envs where Data Lake is not yet configured.
